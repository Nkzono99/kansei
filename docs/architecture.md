# Architecture

Kansei separates a public Python package from a private local instance.

## Public Package

The public repository contains:

- CLI implementation under `src/kansei/cli`
- TOML registry models under `src/kansei/registry`
- provider adapters under `src/kansei/providers`
- dashboard/search/MCP/update helpers
- private-instance templates under `src/kansei/templates`
- repo-local agent guidance under `.agents/skills`
- public JSON schemas under `schemas`
- tests and GitHub Actions workflows

It must not contain private project lists, unpublished research notes, HPC
credentials, API keys, local collaborator details, or real remote paths.

## Private Instance

`kansei init PATH` creates a private control plane:

```text
PATH/
  README.md
  AGENTS.md
  KANSEI.md
  kansei.toml
  projects.toml
  providers.toml
  knowledge/
  dashboards/
  runbooks/
  prompts/
  .codex/
  .kansei/
    manifest.toml
    lock.toml
    state/
    cache/
    logs/
    backups/
    migrations/
```

The instance stores operational metadata and local knowledge. It does not absorb
target project source trees, manuscripts, Slurm outputs, or provider-owned state.

## State Boundaries

The CLI is the source of truth for Kansei state changes. MCP tools and Codex
prompts should call the CLI or mirror CLI behavior rather than write a second
state model.

HarnessOps state is a delegated boundary. Kansei invokes `hops init` and
`hops update-harness` as subprocesses for generated instances, but it does not
directly author `.harnessops/`, `harness-feedback/`, or `harness-lab/`.

Typical flow:

```text
agent or user
  -> kansei CLI / MCP read tool
    -> provider adapter or provider CLI/MCP
      -> target project state
```

## File Ownership

- Managed: lock-tracked files such as `AGENTS.md`, `KANSEI.md`, `.gitignore`,
  `.codex/config.toml`, and template files under `runbooks/_templates` and
  `prompts/_templates`.
- User-owned: registries, knowledge, dashboards, runbooks, prompts, and local
  notes created by the user.
- Generated: files that can be regenerated by an explicit command, such as
  `.codex/config.toml` from `kansei mcp config --write --force`.

See [update-harness](update-harness.md) for the exact update behavior.
